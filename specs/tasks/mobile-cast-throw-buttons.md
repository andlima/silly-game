# Mobile Cast / Throw Buttons

## Goal

Mobile players cannot trigger the two ranged-combat actions, because they are
keyboard-only on desktop (`F` cast, `T` throw) and the action bar only exposes
Food / Use / Wait / Map / Mode / Mute / Help. Add buttons for both, and make
the existing canvas tap-quadrant gesture serve as the direction picker while
the game is in a pending-direction mode.

## Design

### Action-bar buttons

In `index.html`, inside `#action-bar` (currently at lines 309–313), add two
buttons styled identically to the existing ones:

```html
<button id="btn-cast">Cast</button>
<button id="btn-throw">Throw</button>
```

Place them between `btn-food` and `btn-descend`, so the bar reads:
`Food | Cast | Throw | Use | Wait | …`.

Wire click handlers next to the existing ones (around lines 1280–1294):

- `btn-cast` click:
  - If `game.castPending` → dispatch `{ type: 'castCancel' }`.
  - Else → dispatch `{ type: 'cast' }`.
- `btn-throw` click:
  - If `game.throwPending` → dispatch `{ type: 'throwCancel' }`.
  - Else → dispatch `{ type: 'throw' }`.

### Tap-to-aim during pending mode

Update the existing `canvas` `touchend` handler (around lines 1254–1269) so
that when a quadrant tap is detected:

- If `game.castPending` → dispatch `{ type: 'castDir', dir }`.
- Else if `game.throwPending` → dispatch `{ type: 'throwDir', dir }`.
- Else (default) → dispatch `{ type: 'move', dir }` (unchanged).

### Hold-to-repeat guard

Touch hold-to-move (lines 1228–1252) starts auto-firing `move` events after
`HOLD_INITIAL_DELAY`. Suppress this entirely while either pending flag is set,
to avoid an accidental aim from a long press:

- In the `setTimeout` callback (~line 1237): early-return if
  `game.castPending || game.throwPending`.
- In the `setInterval` callback (~line 1245): same guard, and `stopTouchHoldRepeat()`.

### Pending-state visuals

After every render, update button state from `game`:

- `btn-cast`:
  - When `game.castPending` → label `Cancel`, add active style (e.g. yellow background).
  - When no spell held / no charges → `disabled = true` and dimmed.
  - Otherwise → label `Cast`, normal style.
- `btn-throw`: same pattern, gated on having ≥1 throwing dagger in inventory.

A small helper, called from the existing `render()` function, keeps the UI in
sync with whatever the dispatch produced.

Source-of-truth for availability:

- Spell held: check `game.spellHeld` (or whatever field `cast` reads — confirm
  in `src/game.js`; the existing `hud-spell` HUD indicator already reads it).
- Daggers: check the inventory entry the existing `hud-daggers` indicator uses.

Reuse those exact checks so HUD indicator and button-disabled state never
disagree.

CSS for the active/pending state — add a class alongside the existing
`#action-bar button` block:

```css
#action-bar button.active {
  background: #884400;
  color: #ffcc66;
  border-color: #cc8833;
}
#action-bar button:disabled {
  opacity: 0.4;
  cursor: not-allowed;
}
```

### Help overlay update

In `#help-content` (line 302–305), expand the Touch Controls block to list
the new buttons:

```html
<div class="help-row"><span class="help-key">Action buttons</span><span class="help-desc">Food / Cast / Throw / Use / Wait</span></div>
```

(Keep the tap-quadrant row.)

## Changes

| File | What |
|------|------|
| `index.html` | Add `#btn-cast` / `#btn-throw` buttons; wire click + cancel; add `.active` and `:disabled` CSS; update touchend to dispatch `castDir`/`throwDir` while pending; suppress touch hold-repeat while pending; helper to sync button state from `game`; update help overlay Touch Controls row |
| `src/version.js` | Regenerated by `npm run stamp` |

No changes expected in `src/game.js` — the dispatch already supports
`cast`, `castDir`, `castCancel`, `throw`, `throwDir`, `throwCancel`. If a
field/check needed for "is spell available?" or "has daggers?" turns out to
be missing, expose a small read-only helper rather than duplicating internal
shape in the renderer.

## Verify

1. Open `index.html` in a desktop browser with DevTools device emulation set
   to a phone (e.g. iPhone). The action bar shows `Food | Cast | Throw |
   Use | Wait | Map | Mode | 🔊 | ?`.
2. With no spell held, `Cast` is dimmed and unclickable; with a spell held
   (pick up a scroll), it becomes active.
3. Tap `Cast` → message reads "Cast spell — choose direction". Button label
   changes to `Cancel` with active style. Tap a canvas quadrant → spell
   resolves in that direction; button returns to `Cast`.
4. Tap `Cast` then tap `Cancel` → "Spell cancelled." message; no turn
   consumed.
5. Same flow for `Throw` (after picking up throwing daggers): dimmed when
   inventory is empty, active label/style while pending, tap-quadrant
   resolves direction, retap cancels.
6. Long-press canvas while in pending mode does not move the player or fire
   the aim repeatedly.
7. Press `?` on a touch device → help overlay's Touch Controls row lists
   `Food / Cast / Throw / Use / Wait`.
8. Desktop keys (`F`, `T`, arrows, Esc) still work unchanged.
9. `npx jest` passes.
10. Run `npm run stamp` and commit the updated `src/version.js`.
